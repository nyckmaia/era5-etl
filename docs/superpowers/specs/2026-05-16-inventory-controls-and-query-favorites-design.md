# Design — Inventory controls overhaul + Query favorites fix

Date: 2026-05-16
Status: Approved (pending written-spec review)

## Scope

Four user-requested improvements to the web UI:

1. Invert the `/inventory` variable filter so all variables start checked.
2. Replace the `/inventory` point-display controls with a visibility
   checkbox, an opacity slider, and a color picker.
3. Add an `/inventory` hours filter (24-hour checkbox list, all checked).
4. Fix the `/query` "Favoritas" filter showing an empty list.

All four ship together. The SPA is gitignored after build, so the work
is not visible via `era5 ui` until rebuilt (`bun run build` or
`make ui-build`) — the build step is part of this task's definition of
done.

## Affected files

- `web-ui/src/pages/Inventory.tsx` — filter state, controls layout (M1, M2, M3)
- `web-ui/src/components/inventory/InventoryMap.tsx` — opacity/color/visibility props, drop colormap (M2)
- `web-ui/src/lib/api.ts` — add `hour[]` param to `inventory.gridPoints` (M3)
- `web-ui/src/components/query/RightSidebar.tsx` — favorites display/filter (M4)
- `src/era5_etl/web/routes/inventory.py` — accept repeatable `hour` query param (M3)
- `src/era5_etl/storage/coverage.py` — `query_grid_points` hours filter (M3)
- Built SPA assets under `src/era5_etl/web/static/assets/` — regenerated by build

## M1 — Invert variable checkboxes

Today `variableFilter: string[]` is empty by default; empty means "all"
(no `variable` param sent → backend returns every variable's cells). The
checkboxes therefore start unchecked while the map shows everything.

Change to "checked = visible":

- On dataset resolve, seed `variableFilter` to **every** `api_name` of
  `activeDataset.variables` (all checked).
- `changeDataset` re-seeds to all-checked (instead of `[]`).
- Grid-points query parameter logic:
  - all variables checked → send **no** `variable` param (all),
  - a strict non-empty subset → send that subset,
  - **none** checked → render zero points, skip the network query.
- The popover "Todas (limpar seleção)" action becomes a
  "Marcar todas / Desmarcar todas" toggle.
- The button label: all checked → "Todas"; subset → "N selecionada(s)";
  none → "Nenhuma".
- "Limpar filtros" resets variables back to **all checked** (not `[]`).

Rationale: the only behavioral change users see is the checkbox state;
the map still shows all points by default.

## M2 — Professional point controls

Remove the "Pontos" `Visível/Oculto` button and the "Cor"
`Intensidade/Binário` segmented toggle. The intensity colormap (color by
variable count) and `intensityColor()` are deleted — per user decision,
"intensity" is reinterpreted as user-controlled overlay opacity, not a
data encoding.

New controls in the filter bar:

- **"Mostrar pontos"** checkbox → `showPoints: boolean` (default `true`).
- **Opacity slider** `<input type="range" min=0 max=100>` →
  `pointOpacity: number` (default `85`). Shows the numeric % beside it.
- **Color picker** native `<input type="color">` swatch →
  `pointColor: string` hex (default current binary blue `#2864c8`).

All three persist via `useLocalStorage` keyed
`inventory.showPoints` / `inventory.pointOpacity` / `inventory.pointColor`
so the configured view survives reloads.

`InventoryMap` prop changes:

- Remove `colormap` and `totalVars` props.
- Add `pointColor: string` and `pointOpacity: number` (0–100); keep
  `showPoints`.
- `getFillColor` = `[r, g, b, Math.round(opacity/100 * 255)]` where
  `r,g,b` parsed from `pointColor` hex. `ScatterplotLayer.visible` =
  `showPoints`. `updateTriggers.getFillColor` = `[pointColor, pointOpacity]`.
- A tiny hex→`[r,g,b]` helper lives in the component.

## M3 — Hours filter

New filter control modeled on the Variáveis popover, placed after it:

- 24 checkboxes labeled `00:00` … `23:00` (UTC — ERA5 is UTC), all
  checked by default. State: `hourFilter: number[]` seeded `[0..23]`.
- Button label: all → "Todas"; subset → "N selecionada(s)";
  none → "Nenhuma". "Marcar/Desmarcar todas" action inside.
- "Limpar filtros" resets to all 24 checked. `changeDataset` does too.

Filter semantics (user-confirmed, restrictive): a grid point is shown
only if it has **at least one** `(date, variable)` coverage row whose
`hours_mask` contains **every** selected hour, i.e.
`(hours_mask & selectedMask) = selectedMask`.

- `selectedMask` = `Σ (1 << h)` over selected hours.
- All 24 selected → omit the `hour` param entirely (no filter).
- None selected → render zero points, skip the query.

> Consequence: with the default (all 24 checked) the map shows only
> cells with complete 24-hour coverage for at least one (date, variable).
> Cells from partial-hour downloads are hidden by default. This is the
> accepted behavior.

Frontend `api.inventory.gridPoints`: add optional `hour?: number[]`;
append each as a repeated `hour` query param when present.

Backend:

- `routes/inventory.py::grid_points` — add
  `hour: list[int] | None = Query(None, ...)`; pass through.
- `coverage.py::query_grid_points` — add `hours: list[int] | None = None`.
  When non-empty, compute `mask = sum(1 << h for h in hours)` and add a
  clause `(hours_mask & ?) = ?` with params `[mask, mask]` (both UINTEGER
  range). Empty/None → no clause (unchanged behavior). The
  `GROUP BY latitude, longitude` aggregation is unchanged — the clause
  filters rows before aggregation, so a cell appears if any of its rows
  qualifies.

Validation: reject hours outside `0..23` with HTTP 422, consistent with
the existing date validation style.

## M4 — Fix `/query` "Favoritas"

Root cause (verified against `%APPDATA%/era5-etl/query_store.json`): the
store has zero `favorite: true` entries but two **named** entries. The
history row renders `{e.name ? \`★ ${e.name}\` : e.sql}` — a *named*
entry shows a star and looks favorited, but the "Favoritas" filter
filters on the distinct `e.favorite` boolean. The filter is correct; the
star glyph is overloaded across two concepts.

Fix (user decision: "Estrela = só favorito"):

- History row: drop the `★` prefix from named entries. Render the name
  as a plain label (e.g. distinct text style / no star). The `★` is no
  longer used for naming.
- Make the per-entry favorite star button **always visible** (remove the
  `opacity-0 group-hover:opacity-100` gating on it specifically — keep
  the other row actions hover-gated) so favoriting is discoverable.
  Filled amber star = favorite; outline = not.
- The "Favoritas" filter keeps its exact current semantics
  (`!favOnly || e.favorite`). No change to backend or persistence.

Net result: the star a user sees on a row is exactly the flag the filter
uses. Existing named-only entries (`aaa`, `ccc`) will correctly *not*
appear under "Favoritas" until actually starred — which now matches the
visible state.

No data migration: `query_store.json` schema is unchanged.

## Testing

- Backend: extend inventory route tests (FastAPI `TestClient`, no
  network) — `grid-points` with `hour=` params: full set behaves like no
  filter; a subset narrows results; out-of-range hour → 422. Add a
  `CoverageIndex.query_grid_points(hours=...)` unit test using a small
  in-memory/temp coverage DB asserting the `(mask & h)=mask` semantics.
- Frontend: no test harness exists for the React layer in-repo; verify
  by build + manual smoke (variables start all-checked, slider/color/
  visibility affect the layer, hours filter narrows points, Favoritas
  shows only starred). Run the existing `py -3.12 -m pytest` suite to
  confirm no backend regressions.
- Definition of done includes `bun run build` (or `make ui-build`) so
  `era5 ui` serves the changes.

## Out of scope

- GRIB support, manifest/coverage schema changes beyond the read-path
  filter.
- Favorites server model changes (name/favorite stay separate fields).
- Any change to `era5-land` naming or path handling.
